What is a Website?



Websites are a collection of special files that browsers understand and turn into websites we use every day.


Website files include:

  • Code files: HTML, CSS, JavaScript
  • Extra files: images & videos

The Code Files



  • HTML - contains content you see
  • CSS - styles for the site
  • JavaScript - defines behavior of the website

HTML



Contains content you see

  • Defines the structure of the content (navigation, header, footer, lists)
  • Boring to look at by itself
  • Our focus for today

CSS



Styles for the site

  • Decorates HTML content by adding visual appeal
  • Handles layout of content
  • Join us for our next session to dive into CSS!

JavaScript



Defines behavior of the website

  • Muscles that power a website
  • Handles interactions, such as button clicks
  • Gets dynamic content, so you have the most up-to-date information
  • Join us later this year when we cover JavaScript

HTML Syntax

Open/Close Tags



Tags tell the browser how to display the content in the tags


Think of tags as book-ends (In most cases, you have to have a pair: an opening tag and closing tag)


The opening tag has the tag name in angle brackets (< > )

The closing tag has a forward slash (/) before the tag name (</ >)

Open/Closed Tags Continued



Example from MDN:

My cat is very grumpy.

More About Open and Close Tags



Between open/close tags: can be more HTML & line breaks


Example from MDN:


<div>
  <p>
    My cat is very <em>very<em> grumpy.
  </p>
</div>


Some tags are self-closing and don't need a closing tag. That means they can't include more HTML or line breaks. <img src="/images/ryan-gosling.gif" height="460px" />

Required HTML Tags



Every HTML page requires 4 tags in this order:


<!DOCTYPE html>
<html>
  <head>...</head>
  <body>...</body>
</html>

<!DOCTYPE html>



Placed at the very top of the HTML document


Tells the browser what version of HTML is being used


Older versions of doctype were long and ridiculous: <!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Transitional//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd">

<html>



Starts the beginning of the document content


Wrapper of <head> & <body> elements

<head>



First element inside of a <html> tag


Content is invisible to the reader

Gives the browser important infomation about how to interpret the HTML on the page


Other valid tags inside of <head>:

  • <title> - Text that displays in browser tab
  • <meta charset="utf-8"> - Character encoding that can be explicitly set or UTF-8
  • <link> - linked files
    • <link rel="stylesheet" href="/css/style.css"> - Add CSS linked files
    • <script src="/javascript-file.js"></script> - Add JavaScript linked files
  • Metadata for search engines and more

<body>



Element within <html> and after <head>

All the content you want to display in the browser window

HTML Content Tags

HTML Content Tags


Live inside the <body> to give your content structure and meaning


Examples include:

  • Headings, Paragraphs, Ordered/Unordered Lists
  • Hyperlinks, Images, Videos, Audio
  • Tables, Figures
  • Interactive elements such as Buttons, Inputs, Selects
  • Logical groups of related content such as:
    • Navigation, Footer, Address, Quotations, Articles, and Sections
  • Containers that group content such as Divider or Span
  • Content for notes or reminders: <!-- Comment -->

Semantic HTML



Means you organize your content into tags that best convey the meaning of the content.

For example, you can quote Masala Yousafzai using a <span> or semantically correct <q> tag:

<span>We realize the importance of our voices only when we are silenced.</span>
<q cite="https://www.goodreads.com/book/show/17851885-i-am-malala">
We realize the importance of our voices only when we are silenced.
</q>
We realize the importance of our voices only when we are silenced.

We realize the importance of our voices only when we are silenced.

Why is Semantics Important?



Using semantic HTML is a web development best practice because:

  • Necessary for search engines and discoverability (SEO)
  • Necessary for accessibility and creating inclusive websites
  • Helps developers organize their code

HTML Standards & Best Practices



  • Case sensitivity
  • Indent nested tags
    • Every tag should have a close tag that coordinates with an open tag (with the exception of self-closing tags)
  • Use enough line breaks in your code to be readable - don't try to mash code together
  • Be semantic!
<div>

HTML Tag Attributes

Atrributes



Values added to HTML elements that configure the element by changing or adjusting the element's behavior


Set as a property name and value inside the opening tag (including self-closing tags):

attributeName="value here"

There are standard attributes that are:

  • Supported in all HTML elements
  • Only apply to specific elements
  • Custom attributes used for unique needs

Important Attributes



id: supported in all HTML elements to uniquely identify an element

<p id="#hot-toddy"></p>

class: supported in all HTML elements to apply CSS classes for styling

<h1 class="site-header"></h1>

href: specific to hyperlink element

<a href="https://arc.net/">Click here to learn about Arc</a>

src: specific to image element

<img src="https://arc.net/images/arc-logo.png" alt="Arc logo">

data-*: custom attributes that are specific to a website's needs

<div data-menuItem="3">Hot Toddy</div>

Helpful attribute list: MDN Attributes

Helpful Tip About Paths



HTML tags take a path (to an image, file, different page or website)

Paths can be absolute or relative

Relative Paths



Relative paths are relative to the current page

Used for images, files, and links to other pages on the current website domain

  • index.html
  • /about/stars.html
  • /images/arc-logo.png

Absolute Paths



Absolute Paths always include the domain

These are usually page/files on a different website, but can be used for the current website as well

  • http://heygirl.io/
  • https://awkwardfamilyphotos.com/coach/

A11y

What Is It?



  • A11y is a numeronym (type of abbreviation) for accessibility
  • Numeronym: takes the first and last letters of a word & replaces the letters inside with a number representing the letters replaced
    • In this case: 11
    • Another common numeronym is i18n for internationalization
  • Accessibility: making things more usable by members of the disabled community

Why Be Accessible?



  • A better experience for ALL your users
  • Legal obligations (such as ADA) depending on your website
  • Expands your audience

What is POUR?



POUR is a mnemonic for the four principles to increase accessibility

  • Perceivable
  • Operable
  • Understandable
  • Robust

Perceivable



  • Provide text alternatives for any non-text content
  • Pay attention to things like contrast, especially in text
  • Include alternative ways to interact with things like video (captions, video descriptions, sign language, etc)

Operable



  • Allow your website to be operated with keyboard commands, especially tab navigation
  • Consider other input methods
  • Don't include things that could trigger seizures or other reactions
    • Alternatively, include a way to turn them off like "reduce motion"

Understandable



  • Use predictable and consistent navigation & actionable components
  • Include useful feedback and error handling
  • Include help where needed, such as tooltips (remember to make sure they're accessible to those who don't use a mouse!)

Robust



  • Maximize compatibility with current assistive technology

How Can We Be More Accessible?



  • Follow HTML standards and use semantic HTML
  • Include information like "alt" text on images
  • Allow UI controls to be controlled by keyboard (keep it super simple and allow the browser to do its job)
  • Check your website with an accessibility checker:
    • https://www.accessibilitychecker.org/
    • Lighthouse Chrome Tool
  • Use ARIA attributes to provide semantic meaning to content (No ARIA is better than bad ARIA)
    • More info: https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Roles
  • More information about accessible HTML: https://developer.mozilla.org/en-US/docs/Learn/Accessibility/HTML

Introduction to Front-End Architecture

Introduction



Good architecture is a fundamental step in building a house - same applies to applications/websites


Different applications/websites can require different structures

We'll cover basics that can be reused for most of your projects



Of course you can customize as your project requires

What is Front-End Architecture?



Simple definition: Where files & assets go (how they're organized)

Why Is It Important?



Relatively speaking, it's easy to write code, but it's really hard to write maintainable code as:

  • New features are added
  • Size of your team scales

What Does This Look Like?



Mostly conceptual


While the basic pieces/ideas are the same, architecture may vary based on:

  1. Individual/Team Preferences
    • Directory namings such as imgs vs images
    • Organizing by section of site or type of asset, such as app/about/css vs app/css/about
  2. Project Requirements
    • Supporting mobile & desktop? How do you support different styles/assets for various device widths?
  3. Tools/Processes
    • Different web frameworks might have their own recommmendations
